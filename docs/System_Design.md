## 0. 核心設計原則

- **天平只被貨幣影響**，不被任何其他機制直接改變
- **背道者身分不是要被抓住的**，遊戲樂趣在貨幣與天平的陣營爭鬥
- **先修復閉環缺口，再開發新功能**
- **靜態文本存放在 `/src/data/`**，不硬編碼在組件裡

## 4. 背道者系統 [部分實作，需優化]

> ⚠️ 目前有代碼框架但操作需優化，列為待解決清單

### 選拔流程
```
所有玩家完成「去名化道德拷問」適性問問卷
↓
系統標記 is_high_affinity_candidate = true
（此標記僅作為 GM 指定教皇的參考，不觸發任何自動化流程）
↓
GM 在後台手動為每陣營指定 2 人
↓
寫入 identity_role = 'apostate'
身分永久鎖定，章節重置不變更
```

### 每章能力（系統隨機指派A/B/C擇一）

**能力A：背道者的貪婪**
- 從天平隨機抽取 1-3 幣
- 來源陣營隨機（可能抽到自家也可能抽到敵方）
- 不計入當章上限

**能力B：陣營洩漏**
- 背道者可把本陣營一個已完成任務的據點名稱「洩漏」給敵方
- 敵方可看到該據點的同陣營留言內容（原本敵方看不到）
- 執行後背道者自己陣營的人不知道被洩漏
- 直接讓敵方完成一個本陣營據點任務（該任務在敵方顯示「已被敵方解決」）
- 純佔據，不影響天平

**能力C：扒竊**
- 系統隨機從所有玩家中抽一人（排除自己、已失蹤者、NPC、coins=0者）
- 偷取目標貨幣的 10%（無條件捨去，最少1幣）
- 背道者不知道偷了誰，目標不知道是誰偷的
- 僅私人通知雙方當事人
- 消耗章節能力次數

### 背道者專屬UI視覺
```
觸發背道者介面時切換：
背景色：低彩度青色（Cyan）
線條：幾何線條風格
動效：頻閃呼吸燈
背景：噪點紋理
文字：去名化（只說「那裡/這裡」「年長者/年幼者」）
```

## 5. 領主系統 [新增]

### 基本設定
```
每陣營1名領主
每章獲得 3 點惡政點數（章節重置）
有獨立的「金庫」貨幣池（與個人貨幣分開）
有手動編寫公告的介面
```

### 三種惡政（消耗惡政點數）

**徵稅令（消耗1點）**
- 對本陣營所有玩家強制扣1幣進金庫
- 目標 coins = 0 時跳過該玩家（不會變負數）
- 被徵稅玩家當章任務回報多 +1 幣補償
- 自動發布到小道消息

**命名詛咒（消耗1點）**
- 指定本陣營某玩家，強制在匿名代號前加自訂前綴
- 持續本章
- 被詛咒玩家可花 **20幣** 解除（貨幣進入虛空，不進金庫）
- 自動發布到小道消息（顯示匿名代號，不顯示真實OC名）

**荒謬法令（消耗1點）**
- 發布一條無機制效果的陣營公告
- 玩家登入後第一個畫面顯示（可關閉）
- 領主手動編寫，有專屬文字編輯介面
- 自動發布到小道消息

### 金庫玩法

| 門檻 | 行動 | 說明 |
|------|------|------|
| 3幣 | 懸賞令（小額） | 賞1幣，指定玩家完成指定據點任務 |
| 5幣 | 懸賞令（中額） | 賞2幣，指定玩家完成指定據點任務 |
| 8幣 | 詛咒金庫 | 燒掉金庫所有貨幣，隨機一名敵方玩家本章上限-3 |

**懸賞令完成判定**：玩家回報任務時自動比對懸賞條件，符合則自動從金庫轉帳

### 惡政面板 UI/UX 設計 [新增]

#### **1. 陣營化命名**

領主面板需依陣營顯示不同名稱，以符合世界觀設定。

-   **淨塵者 (`Pure`) 陣營：**
    -   領主稱呼：**教皇**
    -   面板標題：**教廷面板**
    -   介面風格：教廷風格
    -   **名詞替換**：
        -   惡政點數 → **信仰忠誠**
        -   金庫 → **聖庫**
        -   徵稅令 → **什一奉獻**
        -   命名詛咒 → **賜福**
        -   荒謬法令 → **教義頒布**
        -   懸賞令 → 懸賞令 (不變)
        -   詛咒金庫 → **審判之火**

-   **濁息者 (`Turbid`) 陣營：**
    -   所有名稱維持原樣（領主、惡政面板、惡政點數等）。

#### **2. 面板實作細節**

**入口**：
-   底部導航列新增「惡政」按鈕，此按鈕僅在 `currentUser.identity_role === 'leader'` 時可見。

**面板結構**：

1.  **基本資訊區**
    -   顯示本章剩餘惡政點數 (`leader_evil_points`，最多3點)。
    -   顯示金庫貨幣數量 (`leader_treasury`)。
    -   帶有對應陣營色的標記。

2.  **三種惡政 (各消耗1點惡政點數)**
    -   **徵稅令**:
        -   按鈕：「發布徵稅令」
        -   確認彈窗：「對本陣營所有玩家強制扣1幣，被徵稅玩家本章任務多+1幣補償」
        -   成功後呼叫 `POST /api/leader/tax`。
    -   **命名詛咒**:
        -   輸入欄：目標玩家的OC名。
        -   輸入欄：自訂的詛咒前綴文字。
        -   按鈕：「發布詛咒」
        -   成功後呼叫 `POST /api/leader/curse`。
    -   **荒謬法令**:
        -   多行文字編輯區，供領主輸入法令內容。
        -   按鈕：「發布法令」
        -   成功後呼叫 `POST /api/leader/law`。

3.  **金庫行動 (消耗金庫貨幣)**
    -   **懸賞令** (金庫需有 3 或 5 幣):
        -   下拉選單或按鈕組，選擇賞金金額 (1幣或2幣)。
        -   輸入欄：目標玩家的OC名。
        -   輸入欄：指定的據點名稱。
        -   按鈕：「發布懸賞」
        -   成功後呼叫 `POST /api/leader/bounty`。
    -   **詛咒金庫** (金庫需有 8 幣):
        -   按鈕：「燒毀金庫」
        -   確認彈窗：「確定要燒掉全部金庫貨幣，對隨機一名敵方玩家本章貨幣上限-3嗎？此操作不可逆。」
        -   成功後呼叫 `POST /api/leader/curse-treasury`。

4.  **本章歷史紀錄**
    -   從 `leader_decrees` 表讀取本章已發布的惡政列表。
    -   清晰展示每條紀錄的類型、發布時間、目標等資訊。

**視覺規格**：
-   領主惡政相關介面元素（如邊框、標題）使用深紅色 `#7f1d1d`。
-   面板應有破損邊框效果 (可使用 `border-dashed` 或不規則的 `box-shadow` 模擬)。
-   整體 UI 風格應營造出沉重、威壓、略帶腐敗的感覺。

### NPC 經營壓力機制 [新增]

**核心概念：**
在每個章節結束時，對商人型 NPC 進行「業績結算」。若未達標，NPC 將進入「經營不善」狀態，暫時失去功能，直到被玩家社群「拯救」為止。

**業績指標：**

| NPC 類型 | 指標 | 說明 |
|---|---|---|
| 雜貨商人 | `sales_total` | 該章節的銷售總額 |
| 寵物商人 | `sales_total` | 該章節的銷售總額 |
| 旅店老闆 | `services_rendered` | 該章節的服務總人次 |

**資料庫變更：**

1.  **`td_users` 表格：**
    *   為 `item_merchant`, `pet_merchant`, `inn_owner` 類型的 NPC 新增以下欄位：
        *   `sales_total` (integer, default 0)
        *   `services_rendered` (integer, default 0)
        *   `is_struggling` (boolean, default false) - 標記是否處於「經營不善」狀態

2.  **`npc_performance` 表格 (新表格)：**

| 欄位 | 類型 | 說明 |
|---|---|---|
| `id` | uuid | 主鍵 |
| `npc_oc` | text | 對應 `td_users.oc_name` |
| `chapter_version` | text | 章節版本 |
| `target_sales` | integer | 該章節的銷售目標金額 |
| `target_services` | integer | 該章節的服務人次目標 |
| `donation_goal` | integer | 進入「經營不善」狀態後的資助目標金額 |
| `current_donation` | integer | 當前已募集的資助金額 |

**API 設計：**

1.  **擴充現有 API：**
    *   `POST /api/npc/merchant/buy`, `POST /api/pets/buy`:
        *   交易成功後，`UPDATE td_users SET sales_total = sales_total + [price] WHERE oc_name = [seller_oc]`
    *   `POST /api/npc/inn/heal`, `POST /api/npc/inn/rescue`:
        *   服務成功後，`UPDATE td_users SET services_rendered = services_rendered + 1 WHERE oc_name = [inn_owner_oc]`

2.  **新增結算 API：**
    *   `POST /api/admin/settle-chapter` (僅限管理員):
        *   遍歷所有商人型 NPC。
        *   比較其 `sales_total` / `services_rendered` 與 `npc_performance` 中設定的目標。
        *   若未達標，`UPDATE td_users SET is_struggling = true`。
        *   重置所有商人 NPC 的 `sales_total` 和 `services_rendered` 為 0。

3.  **新增捐款 API：**
    *   `POST /api/npc/donate`:
        *   **請求：** `{ "donator_oc": "...", "npc_oc": "...", "amount": ... }`
        *   **驗證：**
            *   NPC 必須處於 `is_struggling = true` 狀態。
            *   玩家捐款金額不得超過該章節的個人上限。
        *   **執行：**
            *   扣除玩家貨幣。
            *   `UPDATE npc_performance SET current_donation = current_donation + [amount]`。
            *   若 `current_donation >= donation_goal`，則 `UPDATE td_users SET is_struggling = false`，並重置 `current_donation`。

**前端互動流程：**

*   當 `currentUser.is_struggling` 為 `true` 時：
    *   所有功能按鈕（購買、治療等）變為灰色不可用。
    *   顯示一個「資助」按鈕。
*   點擊「資助」按鈕，彈出一個模態框，顯示：
    *   「資助目標金額」
    *   「目前已募集金額」的進度條
    *   一個輸入框和按鈕，供玩家輸入捐款金額並確認。

## 9. 人販子系統 [重構]

### 移動與聲望
```
每章 movement_points = 10，每步固定扣1
每次執行村民任務 +3 聲望，上限10點
```

### 村民任務
```
人販子抵達據點後可執行
從 /src/data/npc-deliver-texts.json 隨機抽取一段文字顯示
確認後獲得3聲望
```

### 技能表

| 技能 | 聲望消耗 | 效果 |
|------|---------|------|
| 綁架 | 5點 | 目標失蹤6小時 |
| 黑市情報 | 3點 | 查看當前據點本章到訪OC名單 |
| 扒竊 | 8點 | 隨機偷某玩家10%貨幣 |

## 12. 任務系統 [重構]

### 任務類型與獎勵

| 類型 | 限制 | 獎勵 | 計入上限 |
|------|------|------|---------|
| 主線任務 | 每章限接一次 | 3-5幣 | ✅ |
| 支線任務 | 每章限接一次 | 3-5幣 | ✅ |
| 常駐任務 | 每章固定 | 3-5幣 | ✅ |
| 組隊任務 | 達標觸發 | 8-12幣 | ✅ |

每章貨幣上限10幣（組隊/支線/常駐超出部分不累積）

### 回報格式
```
格式：章節-據點名稱-OC名稱
範例：第一章-遺棄醫院-塞理安
```

### 組隊任務流程
```
據點顯示組隊需求（0/3 或 0/1）
↓
玩家點擊「參加」→ is_in_party = true，鎖定無法加入其他組隊
↓
人數達標 → party_slots.status = 'full'
解鎖「▶ 進入」按鈕（非同步，不需同時在線）
↓
各自點擊進入 → 全螢幕文字劇情（打字機效果）
文本來源：/src/data/party-events.json
↓
危險節點出現時：顯示翻牌介面（個人獨立抽）
後端決定鬼牌位置，前端不知道哪張是鬼牌
↓
翻牌動畫 → 個人結果定格
安全牌：無事
鬼牌：扣1HP
↓
結果頁底部出現「前往QQ上傳群像，完成後回報任務」提示
↓
回報任務（正常流程）
組隊夥伴OC名寫入角色卡「見證紀錄」
```

### 組隊超時處理
```
組隊達標後，成員有72小時進入閱讀劇情
超時視為「安然度過」，不扣血，任務仍可回報
管理員可手動解鎖卡死的組隊
```
